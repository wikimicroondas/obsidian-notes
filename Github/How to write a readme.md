# Index
## Readme Driven Development
First step for every project should be to write down the ideas into a document that every member of a team should have access to, that's the `README`. We have to be aware of to whom is meant to be written for, either for final users or contributors.

## Points to be addressed
Not every section must be covered in every readme.

| Section         | Description                                                                              | Resources                    |
| --------------- | ---------------------------------------------------------------------------------------- | ---------------------------- |
| Badges          | Metadata is useful to rapidly check tests status,<br>version or put messages of interest | https://shields.io/          |
| Visuals         | Easy and concise information, provided from<br>GIFs and formatted terminal sessions      | https://asciinema.org/       |
| Installation    | Consider offering a light installation                                                   | https://brew.sh/             |
| Requirements    | Dependencies that has to be installed manually                                           |                              |
| Usage           | Demonstrate a problem solved by the project                                              | examples file                |
| Support         | Any form of organized support                                                            | issue tracker, email address |
| Contributing    | Quick approach on PRs status                                                             | lint, automated tests        |
| Acknowledgement | File to display contributors and authors                                                 |                              |
| License         | Pick a license, any license                                                              | https://choosealicense.com/  |
